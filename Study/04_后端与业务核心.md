# 后端与业务核心

这一份帮你把“真正干活的 Python 部分”拆开。

## 一、后端分层怎么理解

这个项目的后端可以粗分成 4 层：

1. 接口层
2. 配置层
3. 服务层
4. 工具层

## 二、接口层

主要文件：

- `backend/app/main.py`
- `backend/app/routes/extract.py`
- `backend/app/routes/render.py`
- `backend/app/routes/settings.py`

作用：

- 接收前端 HTTP 请求
- 解析请求体
- 分发到对应服务
- 返回 JSON

你可以把这一层理解成：

- “面向网页的外壳”

## 三、配置层

主要文件：

- `backend/app/config.py`
- `backend/configs/replace_map_config.json`
- `backend/configs/logical_rules.json`

作用：

- 提供默认设置
- 保存运行时设置
- 规定 `replace_map` 如何生成
- 规定 `logical` 如何生成

你要特别注意：

- 业务逻辑不全写死在 Python 里
- 有一部分逻辑被放到了 JSON 配置中

这是一种很常见的工程做法。

## 四、服务层

主要文件：

- `backend/app/services/extract_service.py`
- `backend/app/services/replace_map_service.py`
- `backend/app/services/render_service.py`

作用：

- 把多个底层工具串成完整业务流程

你可以把这一层理解成：

- “流程编排层”

### `extract_service.py`

负责：

- 建目录
- 复制文件
- 裁 PDF
- OCR
- 生成 `Defandent`
- 生成 `DemandLetter`
- 生成 `logical`

### `replace_map_service.py`

负责：

- 根据 `Defandent + DemandLetter + logical + replace_map_config`
- 生成最终 `replace_map`

### `render_service.py`

负责：

- 把 `replace_map` 组织成 Word 渲染任务
- 调用底层替换工具输出 docx

## 五、工具层

主要文件：

- `backend/tools/pdf_to_markdown.py`
- `backend/tools/extract_enterprise_report.py`
- `backend/tools/extract_lawyer_letter_infringement.py`
- `backend/tools/build_logical_json.py`
- `backend/tools/word_replace.py`

作用：

- 做具体、单一的技术工作

### `pdf_to_markdown.py`

作用：

- 用 Docling 做 OCR
- 把 PDF 变成 Markdown

输入：

- PDF 路径

输出：

- Markdown 文本

### `extract_enterprise_report.py`

作用：

- 从企业报告 Markdown 里抽结构化字段

特点：

- 主要靠规则，不靠大模型

### `extract_lawyer_letter_infringement.py`

作用：

- 用大模型处理律师函
- 分类段落
- 提取侵权事实
- 改写成适合文书使用的客观表述

### `build_logical_json.py`

作用：

- 根据配置规则自动推导案由、诉请三、不正当竞争事实段

### `word_replace.py`

作用：

- 打开 Word 模板
- 用 `replace_map` 替换占位符
- 输出新 docx

## 六、最重要的数据对象

### `Defandent`

来源：

- 企业报告抽取结果

### `DemandLetter`

来源：

- 律师函抽取结果

### `logical`

来源：

- 规则推导结果

### `replace_map`

来源：

- 上面三个对象再经过配置映射得到

作用：

- 真正用于 Word 占位符替换

## 七、本地文件系统在这个项目里很重要

这个项目不是数据库型项目，核心中间结果都落地到磁盘。

重点目录：

- `backend/storage/uploads`
  - 原始上传
- `backend/storage/cases/<案件名>/input_files`
  - 案件输入
- `backend/storage/cases/<案件名>/ocr_md`
  - OCR Markdown
- `backend/storage/cases/<案件名>/replace`
  - `replace_map.json`、`word_job.json`
- `backend/storage/cases/<案件名>/word_output`
  - 最终 Word 输出

这意味着你学这个项目时，不要只看代码，还要去看这些实际产物。

## 八、这个项目里一个很值得警惕的点

前端 mock 数据和后端真实字段名不完全一致。

比如你会看到前端演示数据里有：

- `企业名称`
- `住所`
- `成立日期`
- `法定代表人`

但后端真实企业报告抽取标准字段是：

- `名称`
- `经营场所`
- `注册日期`
- `经营者`

所以你学习时，要优先相信后端真实工具和真实输出，不要只看前端演示数据。
