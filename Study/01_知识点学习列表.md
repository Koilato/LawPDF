# 知识点学习列表

这一份是总路线图。不要把它当“知识百科”，把它当“过关清单”。

## 第 0 关：先知道项目是干什么的

目标：用一句话说清项目用途。

要看：

- `README.md`
- `backend/workflows/full_pipeline.py`

你要说出来的话：

- 这是一个“网页工作台 + Python 后端”的本地项目
- 输入是企业报告 PDF、律师函 PDF、Word 模板
- 输出是抽取结果 JSON、replace_map、最终 Word 文书

## 第 1 关：理解“前端”和“后端”在这里分别干什么

目标：不要再把前端想成“另一种 Python”。

要看：

- `frontend/src/App.tsx`
- `frontend/src/api.ts`
- `backend/app/main.py`

你要掌握的点：

- 前端负责界面、按钮、状态、展示
- 后端负责 OCR、抽取、规则拼装、Word 替换
- 前后端通过 HTTP 接口连接

过关标准：

- 你能讲出 `App.tsx -> api.ts -> main.py` 这条线

## 第 2 关：最小前端知识

目标：知道 React、TypeScript、Vite 在这个项目里各自扮演什么角色。

要看：

- `frontend/package.json`
- `frontend/vite.config.ts`
- `frontend/src/main.tsx`
- `frontend/src/App.tsx`
- `frontend/src/components/FileInputPanel.tsx`

你要掌握的点：

- `TypeScript` 是前端代码语言
- `React` 是组织页面和状态的方式
- `Vite` 是前端开发服务器和构建工具
- 组件就是页面上的一块块小模块

过关标准：

- 你能说出“为什么浏览器需要这套技术，而不是直接运行 Python”

## 第 3 关：学会追一条真实调用链

目标：只追“执行抽取”这一条，不求全懂。

要看：

- `frontend/src/App.tsx` 里的 `handleExtract`
- `frontend/src/api.ts` 里的 `extractCaseData`
- `backend/app/main.py` 里的 `_handle_extract_request`
- `backend/app/services/extract_service.py` 里的 `extract_case_data`

你要掌握的点：

- 页面点按钮以后发生了什么
- 文件为什么要先转成 base64
- 后端为什么要先把上传内容写回磁盘
- 主流程函数是如何串起多个工具的

过关标准：

- 你能不用看代码，自己画出 4 个文件的调用顺序

## 第 4 关：看懂后端接口层

目标：知道 Python 后端是怎么“像服务端一样”接收前端请求的。

要看：

- `backend/app/main.py`
- `backend/app/routes/extract.py`
- `backend/app/routes/render.py`
- `backend/app/routes/settings.py`
- `backend/app/config.py`

你要掌握的点：

- 这个项目没用 FastAPI/Flask，而是用 `http.server`
- `main.py` 负责路由分发
- `config.py` 负责默认配置和运行时配置
- `routes` 这一层很薄，只是转发到 service

过关标准：

- 你能说出 `/api/extract` 和 `/api/render` 分别进了哪个 Python 函数

## 第 5 关：看懂业务核心流水线

目标：知道“抽取”不是一个函数，而是很多步骤拼起来的。

要看：

- `backend/app/services/extract_service.py`
- `backend/tools/pdf_to_markdown.py`
- `backend/tools/extract_enterprise_report.py`
- `backend/tools/extract_lawyer_letter_infringement.py`
- `backend/tools/build_logical_json.py`

你要掌握的点：

- PDF 先 OCR 成 Markdown
- 企业报告用规则抽取
- 律师函用大模型抽取
- `logical` 由规则配置生成

过关标准：

- 你能说出 `Defandent`、`DemandLetter`、`logical` 分别来自哪里

## 第 6 关：看懂 replace_map 是怎么生成的

目标：知道为什么项目中间要多出一个 `replace_map`。

要看：

- `backend/app/services/replace_map_service.py`
- `backend/configs/replace_map_config.json`
- `frontend/src/utils/replaceMap.ts`

你要掌握的点：

- `replace_map` 是“占位符 -> 最终文本”的字典
- 后端用真实配置生成它
- 前端也有一份本地预览逻辑，但那不是最终真实渲染逻辑

过关标准：

- 你能解释为什么模板替换前必须先有 `replace_map`

## 第 7 关：看懂 Word 是怎么生成的

目标：知道最终 docx 是怎么从模板变出来的。

要看：

- `backend/app/services/render_service.py`
- `backend/tools/word_replace.py`
- `backend/storage/cases/.../replace/word_job.json`
- `backend/storage/cases/.../render_manifest.json`

你要掌握的点：

- `render_service.py` 负责组装渲染任务
- `word_replace.py` 负责真正替换占位符
- 输出文件都落在 `backend/storage/cases/...`

过关标准：

- 你能解释 `word_job.json` 和 `render_manifest.json` 的作用

## 第 8 关：理解项目里的“真实数据”和“演示数据”不是一回事

目标：避免被 mock 数据误导。

要看：

- `frontend/src/mockData.ts`
- `backend/tools/extract_enterprise_report.py`
- `backend/configs/logical_rules.json`
- `backend/configs/replace_map_config.json`

你要掌握的点：

- 前端有演示数据
- 演示数据字段名和后端真实字段名不完全一致
- 这会导致你只看前端时产生错误理解

过关标准：

- 你能举出至少一个字段名不一致的例子
