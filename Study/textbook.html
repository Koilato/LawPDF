﻿# 项目代码理解课本

副标题：用 Python 视角理解这个项目的前端、后端和业务流水线

这份文件是给谁的

这份文件默认读者是：

  • 主要会 Python
  • 对 React、TypeScript、Vite 没什么概念
  • 想先看懂项目，再慢慢补前端知识

这不是“前端大全”，而是围绕当前仓库写的一份项目理解课本。

这份文件怎么用

整份文件只做三件事：

  1. 先把必须学的知识点压缩到最少
  2. 每个知识点只讲到能看懂本项目为止
  3. 每个知识点都绑定到本项目里的具体代码

你读每一节时，只回答 4 个问题：

  1. 这个知识点在项目里是干什么的
  2. 它的原理是什么
  3. 这个项目里对应哪段代码
  4. 如果删掉它，用户会失去什么能力

全书总图

先不要把这个项目看成“前端项目”或者“AI 项目”。

先把它看成一条流水线：

用户在网页点按钮 -> 前端发请求 -> Python 后端处理 -> 结果写到本地文件 -> 网页显示结果

你后面所有学习，都围绕这条线展开。

第一课：先把整个项目看成一条流水线

知识点

项目分层。

原理

一个工程项目通常不是“一个文件做完所有事”，而是分层：

  • 界面层：让用户操作
  • 通信层：把请求发给后端
  • 接口层：后端接请求
  • 服务层：组织业务流程
  • 工具层：做具体技术动作
  • 存储层：保存中间结果和最终输出

本项目代码示例

项目总说明：README.md

整体流水线：backend/workflows/full_pipeline.py

最值得先看的函数：

# backend/workflows/full_pipeline.py
extract_result = extract_case_data(...)
replace_map = build_replace_map(...)
render_result = render_word_from_replace_map(...)

这 3 行已经暴露了项目主干：

  • 先抽取
  • 再生成 replace_map
  • 再渲染 Word

这一课先记住

这个项目的核心不是前端，而是 Python 流水线；前端只是让这条流水线变成网页可操作。

第二课：为什么这个项目需要前端

知识点

浏览器不能直接运行 Python 业务代码。

原理

用户打开网页时，浏览器最擅长运行的是 JavaScript 生态，而不是你的 Python 脚本。

所以只要项目想做成网页，就必须有一层“浏览器能运行的界面代码”。

在这个项目里：

  • Python 负责 OCR、抽取、Word 生成
  • 前端负责上传文件、点按钮、显示结果

本项目代码示例

前端依赖：frontend/package.json

{
  "dependencies": {
    "react": "^18.3.1",
    "antd": "^5.27.1"
  }
}

前端入口：frontend/src/main.tsx

ReactDOM.createRoot(document.getElementById('root')!).render(
  <React.StrictMode>
    <App />
  </React.StrictMode>,
);

这一课先记住

前端不是来替代 Python 的，而是来给 Python 做一个网页控制面板。

第三课：TypeScript 是什么

知识点

TypeScript 是前端代码语言。

原理

你可以先把 TypeScript 理解成：

  • 语法上接近 JavaScript
  • 但会强制你把数据结构写清楚

它的作用很像你在 Python 里给函数参数、返回值、对象结构加类型提示，只不过更严格。

本项目代码示例

类型定义：frontend/src/types.ts

export interface AppSettings {
  apiUrl: string;
  apiKey: string;
  model: string;
  targetKeyword: string;
}

接口返回结构：

export interface ExtractResponse {
  case_name: string;
  markdowns: {
    enterprise_report: string;
    lawyer_letter: string;
  };
  Defandent: JsonObject;
}

这一课先记住

TypeScript 的核心作用不是“炫技”，而是把对象结构写清楚，减少前后端联调时传错字段的概率。

第四课：React 是什么

知识点

React 是组织网页界面的方式。

原理

React 最核心的思想只有一句：

页面 = 当前状态的结果

状态变了，页面就重新更新。

比如：

  • 你还没上传文件时，页面显示“未选择”
  • 上传后，页面显示文件名
  • 后端返回结果后，页面显示 OCR 文本和 JSON

本项目代码示例

页面总控：frontend/src/App.tsx

const [selectedFiles, setSelectedFiles] = useState({
  enterpriseReportPdf: null as File | null,
  lawyerLetterPdf: null as File | null,
  templateFile: null as File | null,
});

这段代码说明：

  • 页面里保存了“当前选中的文件”这个状态
  • 只要这个状态变了，页面显示也会跟着变

这一课先记住

React 你现在不用系统学，只要把它理解成“用状态驱动页面更新”的框架就够了。

第五课：组件是什么

知识点

组件就是页面上的一块功能模块。

原理

为了不让所有界面代码都堆在一个文件里，React 会把页面拆成多个组件。

每个组件通常只负责一小块界面。

本项目代码示例

在 frontend/src/App.tsx 里你会看到：

<FileInputPanel files={selectedFiles} onFileChange={handleFileChange} />
<SettingsPanel settings={settings} onChange={handleSettingChange} />
<OutputPanel logs={logs} output={output} />

对应文件：

  • frontend/src/components/FileInputPanel.tsx
  • frontend/src/components/SettingsPanel.tsx
  • frontend/src/components/OutputPanel.tsx

这一课先记住

组件不是“新的技术概念”，你可以把它理解成“拆出来的页面子函数”。

第六课：页面状态为什么重要

知识点

状态是前端里最重要的东西。

原理

前端页面并不是一张静态图片，它会跟着用户操作不断变化。

这些变化都要靠状态保存：

  • 文件是否选中
  • 是否正在抽取
  • 后端是否已连接
  • 返回的结果是什么

本项目代码示例

frontend/src/App.tsx

const [isBackendReady, setIsBackendReady] = useState(false);
const [isExtracting, setIsExtracting] = useState(false);
const [defandent, setDefandent] = useState<JsonObject>(mockDefandent as JsonObject);
const [output, setOutput] = useState<OutputSummary>({ status: 'idle' });

这一课先记住

如果你不知道一个页面为什么会变，就先找这个页面把哪些东西放进了状态里。

第七课：前端怎样请求后端

知识点

前端通过 HTTP 请求调用 Python 后端。

原理

网页不能直接调用 Python 函数，所以前端只能通过网络请求的方式，把数据发给本地后端服务。

在这个项目里，请求格式是 JSON。

本项目代码示例

请求封装文件：frontend/src/api.ts

const response = await fetch(`${API_BASE_URL}/api/extract`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(payload),
});

这段代码的意思是：

  • 请求地址是 /api/extract
  • 请求方法是 POST
  • 请求体是 JSON

这一课先记住

在这个项目里，前端和后端的连接点不是“直接 import Python”，而是 /api/... 接口。

第八课：为什么前端要把文件转成 base64

知识点

浏览器里的文件对象不能直接原样塞进 JSON。

原理

用户在网页里选中的文件，前端拿到的是浏览器里的 File 对象。

为了通过 JSON 发给后端，需要先把文件内容转成字符串。这个项目选择了 base64。

本项目代码示例

frontend/src/api.ts

async function serializeFile(file: File): Promise<UploadPayload> {
  return {
    name: file.name,
    contentBase64: await fileToBase64(file),
  };
}

这一课先记住

base64 不是业务逻辑，它只是“把文件内容装进 JSON 里传输”的技术手段。

第九课：Vite 是什么

知识点

Vite 是前端开发服务器和构建工具。

原理

你写的 ts/tsx 代码浏览器不能直接原样理解，所以需要一个工具把它跑起来、编译起来。

Vite 在开发时还会做一件很重要的事：接口代理。

本项目代码示例

frontend/vite.config.ts

proxy: {
  '/api': {
    target: 'http://127.0.0.1:8000',
    changeOrigin: true,
  },
},

这段代码表示：

  • 前端页面运行在 5173
  • 当前端请求 /api/... 时，Vite 会转发给 8000 的 Python 后端

这一课先记住

Vite 在这个项目里最重要的价值，是让前端和本地 Python 后端顺滑联调。

第十课：后端不是 Flask 或 FastAPI，而是 Python 标准库服务

知识点

这个项目的后端用的是 http.server。

原理

后端服务的本质是：

  • 接收请求
  • 看请求路径
  • 调用对应 Python 函数
  • 返回 JSON 或文件

很多项目用 Flask/FastAPI 做这件事，但这个项目直接用了标准库。

本项目代码示例

backend/app/main.py

class CasePipelineRequestHandler(BaseHTTPRequestHandler):
    def do_POST(self) -> None:
        if parsed.path == '/api/extract':
            _send_json(self, _handle_extract_request(payload))

这一课先记住

不要被框架名词困住。无论是 FastAPI 还是标准库，接口层本质都是“路径分发到函数”。

第十一课：后端接口层到底负责什么

知识点

接口层负责“接”和“转”，不负责真正业务处理。

原理

一个清晰的工程会把：

  • 接口处理
  • 业务处理

分开写。这样后端主入口不会变成大杂烩。

本项目代码示例

backend/app/main.py 中：

if parsed.path == '/api/extract':
    _send_json(self, _handle_extract_request(payload))

而真正业务继续下沉到：

  • backend/app/services/extract_service.py
  • backend/app/services/render_service.py

这一课先记住

如果你在后端入口里看到了很长的业务逻辑，那通常说明分层不好。这个项目基本把业务下沉到了 service 层。

第十二课：配置为什么单独放出来

知识点

配置和业务代码是两回事。

原理

同一套代码在不同环境、不同案件下可能需要不同参数，比如：

  • API 地址
  • API Key
  • 模型名
  • 目标关键词
  • 图片对齐方式

这些不应该硬编码在流程里。

本项目代码示例

配置入口：backend/app/config.py

def load_runtime_settings() -> dict[str, object]:
    settings = get_builtin_settings()
    settings.update(_read_runtime_settings_file())
    settings.update(_read_env_overrides())
    return _normalize_runtime_settings(settings)

这一课先记住

配置层负责“让同一份代码可以换参数运行”，而不是直接执行业务。

第十三课：抽取主流程才是你最该懂的 Python 文件

知识点

extract_service.py 是“执行抽取”的总控。

原理

业务主流程常常不是自己做所有细节，而是：

  • 先准备输入
  • 再依次调用几个工具
  • 最后收集结果统一返回

本项目代码示例

backend/app/services/extract_service.py

defandent_md_path = ocr_pdf_to_markdown(...)
demand_letter_md_path = ocr_pdf_to_markdown(...)
defandent = extract_enterprise_report(...)
demand_letter = build_lawyer_letter_output(...)
logical = build_logical(...)

这几行已经完整说明主流程：

  • 先 OCR
  • 再分别抽两个 JSON
  • 最后生成 logical

这一课先记住

如果你只能先吃透一个 Python 文件，就先吃透 extract_service.py。

第十四课：OCR 在这个项目里是什么位置

知识点

OCR 不是最终目标，而是中间步骤。

原理

PDF 本身不是很适合直接做文本规则处理，所以项目先把 PDF 转成 Markdown 文本，再对 Markdown 做抽取。

本项目代码示例

backend/tools/pdf_to_markdown.py

pipeline_options.do_ocr = True
result = converter.convert(str(pdf_path))
return result.document.export_to_markdown()

这一课先记住

在这个项目里，OCR 的产物不是最终结果，而是给后续抽取使用的 Markdown。

第十五课：企业报告抽取为什么不用大模型

知识点

结构比较稳定的文本，优先用规则抽取。

原理

如果一份文档格式相对固定，比如企业报告里常见“名称：”“统一社会信用代码：”这种字段，那用规则抽取通常更稳定、更便宜。

本项目代码示例

backend/tools/extract_enterprise_report.py

STANDARD_FIELDS = [
    "统一社会信用代码",
    "名称",
    "经营者",
    "注册日期",
]

以及：

standardized_fields = map_raw_kv_to_standard_fields(raw_kv, DEFAULT_FIELD_MAPPING)
return build_output(standardized_fields, source_name)

这一课先记住

企业报告抽取更像“规则解析器”，不是“AI 对话器”。

第十六课：律师函抽取为什么用了大模型

知识点

半结构化、语义更强的文本，更适合大模型参与。

原理

律师函不像企业报告那样整齐，它更像自然语言段落。

所以项目采用了：

  1. 先切段
  2. 再让模型分类哪些段落是侵权事实
  3. 再让模型提炼客观事实
  4. 再改写成适合文书的表述

本项目代码示例

backend/tools/extract_lawyer_letter_infringement.py

classified = classify_paragraphs(paragraphs, api_url, api_key, model, debug_store)
selected = [item for item in classified if item["category"] == "侵权事实"]
fact_components = extract_fact_components(selected, api_url, api_key, model, debug_store)
rewritten = rewrite_infringement_facts(fact_components, input_path.name, api_url, api_key, model, debug_store)

这一课先记住

这个项目不是“整个文档扔给模型就完事”，而是把大模型嵌在明确的步骤里使用。

第十七课：logical 是什么

知识点

logical 是规则推导出来的中间字段。

原理

项目里有些字段不是从原文直接抄出来的，而是根据条件推导出来的，比如：

  • 案由
  • 诉讼请求三
  • 不正当竞争事实段

本项目代码示例

规则配置：backend/configs/logical_rules.json

"案由": {
  "when": "企业名称包含目标关键词",
  "true": {"mode": "literal", "value": "侵害商标权及不正当竞争纠纷"},
  "false": {"mode": "literal", "value": "侵害商标权纠纷"}
}

生成函数：backend/tools/build_logical_json.py

conditions = {str(name): _evaluate_condition(dict(rule), sources, params) for name, rule in ...}

这一课先记住

logical 不是原始抽取结果，它是“基于抽取结果进一步推理出来的结果”。

第十八课：replace_map 为什么是关键中间层

知识点

replace_map 是连接“抽取结果”和“Word 模板”的桥。

原理

Word 模板不认识 Defandent、DemandLetter、logical 这些对象，它只认识占位符。

所以必须先把结构化结果转换成：

占位符 -> 最终文本

本项目代码示例

配置文件：backend/configs/replace_map_config.json

"[--需要替换的被告--]": {
  "mode": "path",
  "source": "Defandent",
  "path": ["名称", 0, "value"]
}

后端生成：backend/app/services/replace_map_service.py

replace_map[str(keyword)] = _build_mapping_value(dict(mapping), sources, on_missing=on_missing)

这一课先记住

模板替换前，最关键的准备动作就是生成 replace_map。

第十九课：Word 渲染到底在做什么

知识点

Word 渲染本质上就是“打开模板，然后替换占位符”。

原理

后端先把 replace_map 组织成一个渲染任务，再交给底层工具把 docx 改写并另存为新文件。

本项目代码示例

服务层：backend/app/services/render_service.py

job = build_word_job(...)
processed = process_documents(job)

底层工具：backend/tools/word_replace.py

doc = Document(working_src_path)
replace_in_document(doc, job.replacements, job.image_settings, paragraph_align)
doc.save(dst_path)

这一课先记住

“生成 Word”并不是模型生成整篇文书，而是基于模板做替换。

第二十课：为什么这个项目强依赖本地文件系统

知识点

这个项目的大量中间结果都落在磁盘里，不在数据库里。

原理

当项目要处理文件、OCR 文本、中间 JSON、最终 docx 时，最自然的存储方式往往是目录结构，而不是表结构。

本项目代码示例

建目录函数：backend/app/services/extract_service.py

dirs = {
    'input_dir': case_dir / 'input_files',
    'ocr_dir': case_dir / 'ocr_md',
    'replace_dir': case_dir / 'replace',
    'word_output_dir': case_dir / 'word_output',
}

真实产物目录：backend/storage/cases/<案件名>/...

这一课先记住

学这个项目时，不要只盯代码。你必须同时看代码和 backend/storage 里的实际产物。

第二十二课：这个项目里最容易让人困惑的坑

知识点

前端 mock 数据和后端真实字段名不完全一致。

原理

为了演示页面，前端保留了一套 mock 数据；但真实后端输出字段已经发生过调整。

如果你只看前端演示数据，会误以为某些真实字段就是那样命名的。

本项目代码示例

前端演示数据：frontend/src/mockData.ts

你会看到：

  • 企业名称
  • 住所
  • 成立日期
  • 法定代表人

后端真实企业报告抽取：backend/tools/extract_enterprise_report.py

标准字段是：

  • 名称
  • 经营场所
  • 注册日期
  • 经营者

这一课先记住

学习这个项目时，优先相信：

  • 后端工具代码
  • 后端真实输出文件

而不是前端 mock 数据。
