# 模仿模板

这一份是可复用模板。你以后看任何一个不熟悉的项目，都可以照着填。

## 模板 1：项目总览

```text
项目名称：

一句话用途：

用户从哪里开始操作：

最终输出是什么：

前端技术：

后端技术：

核心业务文件：
```

## 模板 2：一条调用链追踪

建议先选一个最明显的用户动作，比如：

- 点击“登录”
- 点击“上传”
- 点击“执行抽取”
- 点击“生成”

然后照着填：

```text
用户动作：

第 1 个文件：
  它负责什么：
  输入是什么：
  输出是什么：
  下一跳是谁：

第 2 个文件：
  它负责什么：
  输入是什么：
  输出是什么：
  下一跳是谁：

第 3 个文件：
  它负责什么：
  输入是什么：
  输出是什么：
  下一跳是谁：

第 4 个文件：
  它负责什么：
  输入是什么：
  输出是什么：
  下一跳是谁：
```

## 模板 3：单个文件拆解

每看一个文件，都不要急着读所有代码。先填这个：

```text
文件路径：

这一层属于：
  前端界面 / 前端请求 / 后端接口 / 服务层 / 工具层 / 配置层

这个文件的角色一句话描述：

最重要的 1 到 3 个函数：

输入：

输出：

谁调用它：

它调用谁：

我现在不懂但不影响继续往下看的点：
```

## 模板 4：单个函数拆解

```text
函数名：

它的作用一句话：

参数是什么：

返回值是什么：

函数内部一共做了几步：
1.
2.
3.
4.

这一函数在整条链路中的位置：
```

## 模板 5：数据流模板

非常适合这种“有很多中间 JSON”的项目。

```text
原始输入：

中间结果 1：
  谁生成：
  长什么样：
  保存在哪里：

中间结果 2：
  谁生成：
  长什么样：
  保存在哪里：

中间结果 3：
  谁生成：
  长什么样：
  保存在哪里：

最终输出：
  谁生成：
  保存在哪里：
```

## 模板 6：防抽象提醒

每次觉得“太抽象”时，强制自己回到这 5 个问题：

1. 这一段代码到底是谁先调用的
2. 它拿到的真实数据是什么
3. 它处理完以后交给谁
4. 它有没有生成磁盘文件或返回 JSON
5. 如果我删掉它，用户会少掉什么功能

## 模板 7：针对当前项目的练习

你现在可以直接照着做这 3 个练习：

### 练习 A

用模板 2 追踪：

- `执行抽取`

必须覆盖：

- `frontend/src/App.tsx`
- `frontend/src/api.ts`
- `backend/app/main.py`
- `backend/app/services/extract_service.py`

### 练习 B

用模板 5 追踪：

- PDF
- OCR Markdown
- `Defandent`
- `DemandLetter`
- `logical`
- `replace_map`
- docx

### 练习 C

用模板 3 拆解：

- `backend/tools/word_replace.py`

要求：

- 只写这个文件的角色
- 不要求一开始就看懂全部细节

## 你最终要形成的学习习惯

不是“碰到陌生技术先查百科”，而是：

1. 先看项目输入输出
2. 再追一条真实调用链
3. 再分层理解文件职责
4. 最后再补技术名词

这是最适合工程代码阅读的方式，尤其适合已经会 Python、但不熟悉前端的人。
